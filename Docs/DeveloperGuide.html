Architecture Overview

The application follows an MVVM-inspired structure tailored for a lightweight utility tool. Views are modular, and business logic is entirely decoupled into static services.

DesktopSupportTool/ ├── App.xaml / .cs # App lifecycle, theme, tray startup ├── DesktopSupportTool.csproj # Project config (.NET 9, WPF + WinForms) ├── GlobalUsings.cs # Shared using statements │ ├── Models/ # Data transfer objects (records / POCOs) │ ├── ActionResult.cs # Success/failure result for fixes │ └── SystemInfo.cs # Hardware/OS data etc... │ ├── Helpers/ # Low-level utilities │ ├── ElevationHelper.cs # UAC elevation detection │ ├── PowerShellRunner.cs # PowerShell / CMD execution engine │ └── RegistryHelper.cs # Registry read/write helpers │ ├── Services/ # Business logic layer │ ├── HealthCheckService.cs # Background system health monitor │ ├── TroubleshootBot.cs # AI Support Agent engine (14 rules) │ └── TroubleshootService.cs # All one-click fix implementations │ ├── Tray/ │ └── TrayIconManager.cs # System tray icon + right-click menu │ └── UI/ ├── MainWindow.xaml / .cs # Shell: sidebar nav + content area └── Views/ # Each tab is a UserControl ├── AgentBotView.xaml # AI chat interface └── TroubleshootView # One-click fix cards

Design Principles

  • Modular views — Each tab is an independent UserControl
  • Service layer — All logic lives in Services/; views only handle UI events
  • Static services — Services are static classes (no DI container needed)
  • Async everywhere — All system operations use Task / async await
  • Per-action elevation — The app runs as a standard user; admin ops invoke elevated: true

How to: Add a New One-Click Fix

Step 1 — Add the Fix Method

Open Services/TroubleshootService.cs and add a static async method.

/// <summary>
/// Fixes the flux capacitor sync issues.
/// </summary>
public static async Task<ActionResult> FixFluxCapacitorAsync()
{
    _log.Info("Troubleshoot", "Running Flux Capacitor fix...");

    // OPTION A: PowerShell (standard user)
    var result = await PowerShellRunner.RunAsync(
        "Restart-Service "Flux" -Force",
        elevated: false,
        timeoutSeconds: 30);

    // OPTION B: CMD command (admin elevation — triggers UAC)
    var adminResult = await PowerShellRunner.RunCmdAsync(
        "flux.exe /reset",
        elevated: true);

    return result.Success
        ? ActionResult.Ok("Fix applied successfully.")
        : ActionResult.Fail($"Fix failed: {result.Output}");
}

Step 2 — Add the UI Card

Open UI/Views/TroubleshootView.xaml and add a card in the appropriate <WrapPanel>.

<!-- Your New Fix -->
<Border Style="{StaticResource Card}" x:Name="CardFlux" MouseLeftButtonUp="Flux_Click">
    <StackPanel>
        <TextBlock FontFamily="{StaticResource IconFont}" Text="&#xE895;" .../>
        
        <StackPanel Orientation="Horizontal">
            <TextBlock Text="Fix Flux Capacitor" .../>
            
            <!-- Include this Badge ONLY if elevated: true -->
            <Border Background="{StaticResource WarningBrush}">
                <TextBlock Text="🛡️ Admin"/>
            </Border>
        </StackPanel>
        
        <TextBlock Text="Resets the flux sync service" .../>
        <TextBlock x:Name="StatusFlux" Text="" .../>
    </StackPanel>
</Border>

Step 3 — Add the Click Handler

Open UI/Views/TroubleshootView.xaml.cs to handle the click.

private async void Flux_Click(object sender, MouseButtonEventArgs e)
{
    // The RunAction helper handles showing the "running…" status, 
    // disabling the card, calling the fix, and showing ✅/❌ result.
    await RunAction(CardFlux, StatusFlux, "Running fix...",
        () => TroubleshootService.FixFluxCapacitorAsync());
}

UAC Elevation Model

This application does NOT require administrator rights to run. It uses a per-action elevation model, ensuring maximum security and compatibility in locked-down enterprise environments.

Execution Flow
  1. App starts as Standard User.
  2. User clicks a standard fix (e.g. "Clear Temp Files").
  3. PowerShell script executed invisibly under the current user's context.
  4. User clicks an admin fix (e.g. "SFC Scan").
  5. PowerShellRunner spawns a new process with Verb = "runas".
  6. Windows displays the UAC prompt overlay.
  7. Elevated background process runs the command and returns output.

Part 2: Mass Deployment Guide

Strategies for packaging and deploying the Desktop Support Tool to enterprise endpoints.

Building the Distributable

The recommended approach is a Self-Contained Single-File EXE. This includes the .NET 9 runtime inside the binary, meaning target PCs need zero prerequisites installed.

# Build Self-Contained EXE (~60 MB)
dotnet publish -c Release --self-contained true -r win-x64 `
    -p:PublishSingleFile=true `
    -p:IncludeNativeLibrariesForSelfExtract=true `
    -p:EnableCompressionInSingleFile=true `
    -o .\publish

Deployment via SCCM / MECM

Step 1: Application Package

Create a Script Installer application. Content location pointing to your network share containing the compiled DesktopSupportTool.exe.

Step 2: Install Command

This script copies the EXE locally and creates a multi-user run key for auto-start.

powershell -ExecutionPolicy Bypass -Command "
    $dest = 'C:\Program Files\DesktopSupportTool';
    New-Item -Path $dest -ItemType Directory -Force;
    Copy-Item '.\DesktopSupportTool.exe' $dest -Force;
    $regPath = 'HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Run';
    Set-ItemProperty -Path $regPath -Name 'DesktopSupportTool' -Value (Join-Path $dest 'DesktopSupportTool.exe');
"

Step 3: Detection Method

File System → File Exists: C:\Program Files\DesktopSupportTool\DesktopSupportTool.exe

Deployment via Microsoft Intune (MEM)

Step 1: Prep the Package

Download the Microsoft Win32 Content Prep Tool. Create an install.ps1 and uninstall.ps1 in a folder next to your EXE, then compile to an .intunewin file.

IntuneWinAppUtil.exe -c .\SourceFolder -s DesktopSupportTool.exe -o .\Output

Step 2: Configuration

  • Install command: powershell -ep bypass -file install.ps1
  • Install behavior: System
  • Requirements: Windows 10 21H2 or later

Deployment via Group Policy (Startup Script)

For domains without SCCM/Intune, use a Computer Startup script. This script automatically checks file modification dates to pull updates from the central share.

# Deploy-SupportTool.ps1
$source = "\\fileserver\Software\DesktopSupportTool\DesktopSupportTool.exe"
$dest = "C:\Program Files\DesktopSupportTool"
$exePath = "$dest\DesktopSupportTool.exe"

# If current version exists, don't overwrite
if (Test-Path $exePath) {
    if ((Get-Item $exePath).LastWriteTime -ge (Get-Item $source).LastWriteTime) { exit 0 }
}

New-Item -Path $dest -ItemType Directory -Force
Copy-Item $source $dest -Force
Set-ItemProperty -Path "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Run" -Name "DesktopSupportTool" -Value $exePath

Security & AppLocker (WDAC)

Application Control If your organization uses strict AppLocker or Windows Defender Application Control (WDAC), you must allowlist the application. 
# AppLocker Publisher Rule Example
New-AppLockerPolicy -RuleType Path `
    -RuleNamePrefix "DesktopSupportTool" `
    -User "Everyone" `
    -AllowWindows `
    -Path "C:\Program Files\DesktopSupportTool\*"
